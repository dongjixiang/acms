**原型需求专用领域规则：**

1. **原型的技术选型是"原型架构"而非"生产架构"**
   - ❌ 绝对禁止问：Vue 还是 React？Angular 还是 Svelte？用什么数据库？后端用 Node 还是 Java？要不要微服务？用哪个云平台？
   - ❌ 绝对禁止问：React + Ant Design、Vue 3 + Element Plus、Angular + Material Design 这类生产框架选项
   - ✅ 应该问：单 HTML 文件还是多文件拆分？Mock 数据用硬编码 JS 还是独立 JSON？纯 CSS 还是用轻量方案？页面切换用刷新还是 hash 路由？
   - ✅ 应该问：图片资源用 Base64 内嵌还是单独文件？

2. **用户已说 "HTML/CSS/JS" 或 "纯前端" 时**，不要再给 "Vue/React/Angular" 选项。基于 HTML/CSS/JS 询问原型级实现细节即可。

3. **典型的原型技术选择题：**
   - 文件结构：单 HTML（全部内嵌）vs 多文件（HTML+CSS+JS 分离）
   - Mock 数据：硬编码在 JS 中 vs 独立 JSON 文件
   - CSS：纯 CSS vs 轻量方案（Tailwind/Pico.css）
   - 导航：简单的页面切换（显示/隐藏 div）vs hash 路由
   - 图片/资源：Base64 内嵌 vs 单独资源文件

3. **关注交互与流程** — 原型澄清的重点是：
   - 页面结构（哪些页面、每个页面有什么）
   - 用户操作路径（从打开到关闭的完整流程）
   - 交互细节（点击/滑动/拖拽/弹窗/表格操作）
   - 数据展示（表格/图表/列表/详情/表单）

4. **验收标准简化** — 原型的验收标准只需要：
   - "所有页面可正常浏览"
   - "交互流程完整（从入口到退出）"
   - "数据展示与设计一致"
   - 不需要量化性能指标（加载时间、帧率等）

5. **scopeIn 颗粒度** — 原型按**页面维度**列出即可：
   - ✅ "首页：展示数据看板，包含统计卡片和趋势图"
   - 不需要拆到组件级

6. **scopeOut（非功能范围）** — 原型默认排除：
   - 后端服务、数据库
   - 用户认证、权限
   - 真实数据接入
   - 响应式/多端适配

7. **拆分建议规则** — 即使需求涉及多个功能域，原型需求通常不分拆，保留为单一需求更易于管理。
