# 约束突破工具箱 — 帮「毫无头绪」的用户从零起步

## 方法论

用户只有一个领域概念，没有任何具体方向。需要从极端的约束条件入手，倒逼出想法。

## 可选方法（按需选 1-2 个）

### 倒计时失效
极端时间约束倒逼出核心功能。

引导语：
> 「假如只剩 24 小时必须交付第一个版本，你会保留哪一个功能？」
> ——用户回答后追问：「那第二重要的呢？24 小时做不到的话放宽到 3 天？」

### 荒谬方案法
给出一个明显错误但符合用户原话的方案，让用户用「修正币」来改。

引导语：
> 「那你听我猜的对不对——按你的描述，这个产品大概是 **[AI 生成一个明显有缺陷的方向]**。你觉得哪里最不对？指出 3 个要改的地方。」
>
> 用户指出的第一个问题通常就是他在意的核心。

### 原型破坏
如果用户需要视觉辅助，才用这个方法。构造一个极简陋的文字描述让他「戳破」。

引导语：
> 「我们想象一个最简单的版本——打开页面就是一个 **[文本框/按钮/输入框]**。点一下会 **[简单动作]**。你觉得这个哪里不对？」

## 产出

```json
{
  "elicited_boundaries": [
    { "dimension": "核心功能", "value": "极简注册", "confidence": 0.7 },
    { "dimension": "最少可用时间", "value": "3 天", "confidence": 0.6 }
  ],
  "spark": "用户对『注册即用』最敏感"
}
```

## 原则

- blank 类型的用户最需要「安全感」——不要让他感到被审问
- 每个方法都要包装成「游戏」或「假设」而不是「需求分析」
- 产出可能很少甚至没有 elicited_boundaries，有 spark（一个灵感线索）就够了
