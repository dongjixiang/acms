# 需求启发方法论（Elicitor Methods）

> **来源**：v0.4 elicitor SKILL 设计文档（2026-06-13）。
> **目的**：把"需求启发师"的工作方法沉淀为可调用的工具箱。

---

## 核心思想

把"帮用户想清楚一个模糊想法"这件事拆成**三段式**：

1. **诊断（Diagnose）** — 判断用户当前处于什么状态（说不细 / 矛盾想法 / 没头绪）
2. **工具箱（Toolbox）** — 根据诊断结果选 1-2 个轻量方法帮用户前进
3. **固化（Solidify）** — 把对话中的发现收拢成可量化的边界

**关键原则**：
- **不替用户做选择** —— AI 只递引导性问题，最终决策权在用户
- **公开诊断** —— 诊断结果要展示给用户（不是后台偷偷算）
- **轻量方法** —— 每个工具箱方法是一句话问题，不是 UI 卡片
- **可反悔** —— 用户改了主意，新答案覆盖旧答案

---

## 诊断三态

### vague（已有一个大致方向，想具体化）
- **特征**：用户能描述"是什么"，但说不清"具体怎么做"
- **典型话术**：「我想做一个……」「大概就是……」「类似 XX 但更……」
- **判断信号**：描述里有动词但缺少量化对象

### conflicted（有好几个想法在犹豫）
- **特征**：提到两种以上可能性，或描述中自相矛盾
- **典型话术**：「我在想是 A 还是 B」「既要……又要……」
- **判断信号**：一句话里出现"或者"、"还是"、"既要 X 又要 Y"

### blank（完全开放，没头绪）
- **特征**：只有模糊的领域，没有具体方向
- **典型话术**：「我想做点东西」「有没有好点子」「帮我想想」
- **判断信号**：描述 < 30 字、没动词、没具体对象

**置信度阈值**：confidence < 0.6 时默认走 vague（宁松勿严）。

## 工具箱方法集

### vague 工具箱（具象化）

| 方法 | 一句话 | 适用场景 |
|------|--------|----------|
| 场景压缩 | "用一句话告诉我，这个方案解决的是哪个『就那三秒钟』的场景？" | 用户描述抽象，没具体场景 |
| 极端对比 | "如果必须牺牲一项：速度、颜值、成本，你放弃哪个？" | 用户在意多维度但优先级不清 |
| 视觉锚点 | "三种风格里你最接近哪个？哪个地方必须改？" | 用户在意视觉、需要具象化感受 |

### conflicted 工具箱（逆向筛选）

| 方法 | 一句话 | 适用场景 |
|------|--------|----------|
| 反向清单 | "这个方案绝对不许出现的三个特点是什么？" | 用户列了一堆"想要的"，需要反向 |
| 失败预演 | "如果这个项目半年后失败了，最可能的原因是什么？怎么避免？" | 用户没意识到隐藏风险 |
| 替身视角 | "假设你最好的朋友遇到同样场景，你会建议他优先关注什么？" | 用户陷入自己的视角盲区 |

### blank 工具箱（约束突破）

| 方法 | 一句话 | 适用场景 |
|------|--------|----------|
| 倒计时失效 | "假如只剩 24 小时必须交付第一个版本，你会保留哪一个功能？" | 用户没核心 |
| 荒谬方案法 | "按你的描述，这个产品大概是 [AI 故意给个有缺陷方向]，哪里最不对？" | 用户描述太泛，需要反向刺激 |
| 原型破坏 | "我们想象最简版——打开页面就是一个 [文本框]，点一下会 [动作]，哪里不对？" | 用户没具体形态 |

## 固化产出格式

每轮工具箱对话后，提取出的可量化边界：

```json
{
  "elicited_boundaries": [
    { "dimension": "速度", "value": "不超过 2 秒", "confidence": 0.8, "source": "极端对比" }
  ],
  "tradeoff_points": [
    { "dimension": "速度 vs 安全", "user_stance": "倾向安全但犹豫" }
  ],
  "spark": "用户对『注册即用』最敏感"  // blank 类型特有
}
```

**固化原则**：
- 只写用户自己表达过的，不要 AI 脑补
- tradeoff_point 比 boundary 更值钱——用户犹豫的地方就是真正的设计决策点
- summary 用自然语言写给用户看

## 与 ACMS 的映射

| Elicitor 路径 | ACMS 现有对应 | 备注 |
|--------------|--------------|------|
| vague → 具象化 | tradeoff / decision_tree | 现有工具，触发时机后移 |
| conflicted → 逆向筛选 | scenarios / diagnosis | 现有工具，触发时机后移 |
| blank → 约束突破 | **无对应** | 新能力，不走卡片，纯对话 |

**assist 卡片定位变化**：
- 当前：5 种 assist 卡片是"第一道菜"
- 改后：诊断对话是"第一道菜"，assist 卡片是"深度用户"才看到的"点心"

## 设计哲学对齐（4 旅程）

- **「放」= AI 先干到 80%**：诊断在后台 run，AI 把判断结果转成一句话引导问题递给用户
- **「收」= 用户做最后 20% 决策**：用户回答引导问题，决定下一轮工具箱方法
- **「让用户喜欢使用」**：认知负担从"理解 5 种工具有啥区别"降到"回答一句话问题"
